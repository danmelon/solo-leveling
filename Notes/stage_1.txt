Foundation layer:

    db.py > creates all tables on first run, including gym log, meditation log, objectives and trees.
    models.py > all database read/write logic, XP levelling (with scaling difficulty).
    main.py > launches everything, seeds data, dark theme.

Health Tree:

    trees/health.py > seeds 9 objectives across 3 tiers, and all 5 meditation types with full descriptions
    ui/health_tab.py > full UI with gym calendar (click to toggle days), streak counter, meditation logger with info dialogs and an objective checklist that awards XP and triggers level-up popups.


Explanation of the files:

db.py - database setup:
    > The first file that runs when we launch the app. 
    > init_db() creates all the tables if they do not already exist.
        > IF NOT EXIST clause means it is safe to call every time the app starts without wiping our data.

    get_connection():
        > Small helper that every other file calls whenever it needs to talk to the database.
        > The key line is conn.row_factory = sqlite3.Row
            > Without this, SQLite returns plain tuples, meaning we'd have to access data by index like row[0].
            > With it, we can access columns by name like row["date"] which makes the code much more readable.
            > PRAGMA foreign_keys = ON tells SQLite to actually enforce the relationships between tables.

    Schema:
        > Has five tables:
            > trees (one row per skill tree, storing level, XP)
            > objectives (goals that belong to a tree)
            > gmy_log (one row per day you went to the gym)
            > meditation_types (the 5 types with their descriptions)
            > meditation_log (each individual session you log)

models.py - Data Access layer:
    > The brain of the app. No UI lives here, it's purely ask the database to do something, get data back or write something to the database.
    > Every UI component calls functions from here rather than writing SQL itself. Keeps things clean.
        > If we ever want to change how data is stored, we only have to do it in one place.
    
    XP & Leveling:
        > add_xp() - fetches the tree, adds the XP then runs a while loop rather than an if.
            > We do while loop so that if we ever earn enough XP to level up more than once in one go, it handles all the level-ups in sequence.
            > Each time we level up, it scales xp_per_level by 1.2x so level 1 needs 100xp, level 2 needs 120xp, level 3 needs 144xp
            > It returns the updated tree dict plus a levelled_up boolean so the UI knows whether to show the level-up popup.

    Completing objectives:
        > complete_objective() marks the obj done then calls add_xp().
        > Checks if obj and not obj["completed"] before doing anything, so double checking an already completed objective doesn't award XP twice.

    Gym streak:
        > get_gym_streak() fetches all our gym logs ordered by most recent first, then walks backwards through them day by day starting from today.
        > As long as each date matches the expected "next day back", the streak counter goes up.
        > The moment there's a gap it stops.
        > The date arithmetic handles month boundaries... block at the bottom of the code calcs what day yesterday was when at the 1st of the month.

    Gym log toggle:
        > log_gym() uses try/except around the INSERT rather than checking first.
        > gym_log table has UNIQUE on the date column so inserting the same date twice throws an exception, which gets caught and returns FALSE.

trees/health.py - Seed Data:
    > File just contains the content.
        > List of meditation types and objectives, along with a seed() function that inserts them into the database.
        > ON CONFLICT(name) DO NOTHING line makes it safe to call every time the app start:
            > If the data already exists it silently skips the insert.
        > For objectives:
            > Rather than using that pattern, it checks if existing == 0 first and only seeds if the table is completely empty for that tree.

        > Objectives designed in three tiers:
            > Habit building, cosistency, and mastery.
            > App does not formally group them (yet) - we use the sort_order column.

        
ui/health_tab.py - The Health UI:
    XPBar:
        > Reads the tree from the database and renders the level number and a progress bar.
        > The bar width is calculated as a percentage of xp / xp_per_lvl, then scaled to the pixel width of the container.
        > refresh() is called by other components every time XP changes.

    GymCalendar:
        > Builds monthly grid of buttons using Python's built-in calendar.monthcalendar().
            > Retuns a list of weeks, each being a list of 7 day numbers with 0 for days outside the month.
            > Each button is either green (logged), highlighted (today), or neutral.
            > Clicking a button calls _toggle() which adds or removes the gym log for that date.
                > Then calls refresh() to redraw the whole grid.
            > Prev/next month buttons just change _year & _month attributes and call refresh().
    
    MeditationPanel:
        > Has a dropdown (populated from the meditation types in db), duration spinner, optional notes field.
        > When you log a session it calls models.log_meditation() and also directly awards 10XP via models.add_xp().
            > This is separate from the objectives system, meaning we earn small XP just for meditating regardless of which milestone we're on.
        > Info button opens MeditationInfoDialog which displays summary of text for whichever type is selected.
    
    ObjectivesPanel:
        > Renders each objective as a row with a checkbox button, title, description and XP reward.
        > _toggle() method checks whether the objective is already completed
            > If it is, clicking uncompletes it & no XP refund (currently).
            > If it is newly completed, it calls complete_objective() and checks returned dict for levelled_up to decide whether to show the popup.
    
    HealthTab:
        > Just assembles all of these widgets into a two-column layout inside a scroll area and calls xp_bar.refresh() on load.


ui/main_window.py - App Shell:
    > Sets up sidebar & QStackedWidget, which is PyQt's way of having multiple pages, where only one is visible at a time.
    > Each nav button calls _switch(index), which tells the stack which page to show and updates which button appears as selected.
    > Tabs are loaded in _load_tabs() - currently only Health is built out, while the others remain as placeholder.

main.py - Entry Point:
    > db.init_db() creates tables.
    > seed_all_trees() populates them with starting data.
    > PyQt app launches.
        > If we tried to seed before the tables existed it would crash.
